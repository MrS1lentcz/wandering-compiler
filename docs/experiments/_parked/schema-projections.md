# Experiment: Schema Projections & Project Structure — how authoring would look

The goal of this file is to **visualize what and how a developer writes** when defining
a new domain object. Each experiment models the same use case (a product in an e-shop),
so authoring styles can be compared side by side.

Files are **glued together into one document** with large separators — in a real project
each block lives in its own file per the structure below.

## Project structure (reference)

```
w17/
├── common/                         # shared enums, value types, template models
└── domains/
    └── <domain>/                   # DB-isolated unit (3rd-party-like contract outward)
        ├── apis/                   # 1+ facades — external SYNC contract
        ├── services/               # 1+ standard gRPC — business logic, module composition
        ├── events/                 # external ASYNC contract (event definitions)
        └── modules/
            └── <module>/           # internal data organization (FK/query across modules OK)
                ├── models/         # tables
                └── queries/        # storage methods → auto-aggregated into StorageService
```

- **Storage service:** always 1 per domain, auto-derived from all queries across modules.
- **Standard services:** 1+ per domain; developer decides based on separation of concerns.
- **Facades:** 1+ per domain, usually split by consumer (public / admin / internal).
- **Extensions** (`w17.schema.projection`, `w17.field`, ...) live in the authoring layer.
  The generator expands them into clean proto under `gen/proto/` before `protoc` runs —
  the user never sees or edits that layer.

```
 The developer writes:
   1) one DB model            (the single source of truth about the table)
```

### `w17/domains/catalog/modules/products/models/product.proto` — the table

```protobuf
syntax = "proto3";
package w17.catalog.products;

import "w17/db.proto";
import "w17/field.proto";
import "w17/validate.proto";

message Product {
  option (w17.db.table) = {
    name:       "products"
    timestamps: true          // generates created_at, updated_at
    soft_delete: true         // generates deleted_at + filtering in queries
    indexes: [
      { fields: ["slug"], unique: true },
      { fields: ["category_id", "is_active"] }
    ]
  };

  string id               = 1  [(w17.field) = { pk: true, immutable: true }];
  string slug             = 2  [(w17.validate) = { min_len: 1, max_len: 120 }];
  string name             = 3  [(w17.validate) = { min_len: 1, max_len: 255 }];
  string description      = 4;
  double price_cents      = 5  [(w17.validate) = { min: 0 }];
  string category_id      = 6  [(w17.field) = { fk: "w17.catalog.categories.Category.id" }];
  int32  stock_quantity   = 7  [(w17.validate) = { min: 0 }];
  bool   is_active        = 8;
  string internal_notes   = 9;   // internal, must never leak out
  double cost_price_cents = 10;  // internal, must never leak out
}
```

### `w17/domains/catalog/apis/public.proto` — public facade (what the customer sees)

```protobuf
syntax = "proto3";
package w17.catalog.api.public;

import "w17/schema.proto";
import "w17/field.proto";

// ----------------------------------------------------------------------------
// Public response — "what the customer sees"
// ----------------------------------------------------------------------------
message Product {
  option (w17.schema.projection) = {
    from: "w17.catalog.products.Product"
    omit: ["internal_notes", "cost_price_cents"]
  };
  // Fields are NOT retyped here — the generator pulls them from the model.

  // Added computed field — not in DB; evaluated in SQL or Go.
  bool is_in_stock = 100 [(w17.field) = {
    computed: "stock_quantity > 0"
  }];
}

// ----------------------------------------------------------------------------
// Create request — "what the client sends"
// ----------------------------------------------------------------------------
message CreateProductRequest {
  option (w17.schema.projection) = {
    from: "w17.catalog.products.Product"
    // id, created_at, updated_at, deleted_at are automatically absent from the
    // request — they are marked system-managed on the model.
    omit: ["internal_notes", "cost_price_cents"]
  };
}

// ----------------------------------------------------------------------------
// Update request — partial update
// ----------------------------------------------------------------------------
message UpdateProductRequest {
  option (w17.schema.projection) = {
    from:    "w17.catalog.products.Product"
    partial: true                    // every field wrapped optional
    omit:    ["internal_notes", "cost_price_cents"]
  };
  // Field-level override: slug cannot be changed after creation.
  string slug = 2 [(w17.field.projection) = { readonly: true }];
}
```

### What the developer gets without writing code

| Generated automatically                             | Source                                      |
|-----------------------------------------------------|---------------------------------------------|
| SQL migration for `products` table                  | `modules/products/models/product.proto`     |
| DB gRPC CRUD (Create/Get/Update/Delete/List)        | DB model                                    |
| `CREATE VIEW product_public` with `is_in_stock`     | projection `api.public.Product`             |
| REST endpoints + OpenAPI                            | facade messages + protobridge               |
| TS client with proper type variance                 | facade messages                             |
| Django-style admin list/detail/edit for `Product`   | `api.admin.ProductAdmin` + UI metadata      |
| Input validation (`CreateProductRequest`)           | `w17.validate` on the DB model              |

```protobuf
message ProductAdmin {
  string id               = 1;   // again
  string slug             = 2;   // again
  string name             = 3;   // again
  string description      = 4;   // again
  double price_cents      = 5;   // again
  string category_id      = 6;   // again

```

**Duplication for a 10-field object with 4 projections:** ~30 retypes. When a type
changes or a column is added the developer has to edit `N` places — the compiler
offers no help because the proto messages have no declared relationship.

```
================================================================================
 EXPERIMENT C — EDGE CASES AND THE ESCAPE HATCH
================================================================================

 Covers:
   1) computed field evaluated in DB (SQL view)
   2) computed field evaluated in Go (business logic)
   3) field rename in a projection
   4) escape hatch — no projection, plain message written by hand
================================================================================
```

```protobuf
syntax = "proto3";
package w17.catalog.api.public;

import "w17/schema.proto";
import "w17/field.proto";

// ----------------------------------------------------------------------------
// (1) DB-computed field + SQL view
// ----------------------------------------------------------------------------
message ProductListItem {
  option (w17.schema.projection) = {
    from:      "w17.catalog.products.Product"
    omit:      ["description", "internal_notes", "cost_price_cents"]
    emit_view: "product_list_items"   // -> CREATE VIEW product_list_items AS ...
  };

  // Computed in SQL — the generator adds the expression to the CREATE VIEW.
  double margin_cents = 100 [(w17.field) = {
    computed: "price_cents - cost_price_cents"
    sql_only: true
  }];
}

// ----------------------------------------------------------------------------
// (2) Go-computed field (not in SQL; evaluated in handler/mapper)
// ----------------------------------------------------------------------------
message ProductCard {
  option (w17.schema.projection) = {
    from: "w17.catalog.products.Product"
    pick: ["id", "name", "price_cents"]   // opposite of omit — only listed
  };

  string display_price = 100 [(w17.field) = {
    computed_go: "fmt.Sprintf(\"%.2f Kc\", float64(src.PriceCents)/100)"
  }];
}

// ----------------------------------------------------------------------------
// (3) Rename — different name in API than in DB
// ----------------------------------------------------------------------------
message ProductPublic {
  option (w17.schema.projection) = { from: "w17.catalog.products.Product" };

  // Field #3 in the model is named `name`. Here we expose it as `title`.
  string title = 3 [(w17.field.projection) = { source: "name" }];
}

// ----------------------------------------------------------------------------
// (4) Escape hatch — no projection, fully hand-written message
//     Used when the shape does not correspond to any DB model
//     (aggregated statistics, multi-source response, etc.).
// ----------------------------------------------------------------------------
message CheckoutSummary {
  string   order_id        = 1;
  int32    item_count      = 2;
  double   subtotal_cents  = 3;
  double   tax_cents       = 4;
  double   total_cents     = 5;
  repeated string warnings = 6;
}
```

```protobuf
package w17.catalog.products;

import "w17/db.proto";
import "w17/field.proto";
import "w17/validate.proto";
import "w17/common/types/money.proto";   // ← reference into common/

message Product {
  option (w17.db.table) = {

    soft_delete: true
    indexes: [
      { fields: ["slug"], unique: true },
      { fields: ["category_id", "is_active"] }
    ]
  };

  string id               = 1 [(w17.field) = { pk: true, immutable: true }];
  string slug             = 2 [(w17.validate) = { min_len: 1, max_len: 120 }];
  string name             = 3 [(w17.validate) = { min_len: 1, max_len: 255 }];
  string description      = 4;

  // Cross-module FK — same domain = same DB, so it is allowed.
  string category_id      = 5 [(w17.field) = {
    fk: "w17.catalog.categories.Category.id"
  }];

  // Cross-common reference — shared value type from common/.
  // Each domain materializes its own columns; only the SHAPE is shared.
  w17.common.types.Money price = 6;

  int32 stock_quantity = 7 [(w17.validate) = { min: 0 }];
  bool  is_active      = 8;
}
```

### `w17/domains/catalog/modules/products/queries/list_active_by_category.qd`

```
# Storage method → auto-aggregated into StorageService.ListActiveByCategory.
# Used by the standard service (layer 2). Never called by clients directly.

query ListActiveByCategory(category_id: string, limit: int32 = 50) -> [Product] {
  from: Product
  where: {
    Product.category_id: $category_id
    Product.is_active:   true
  }
  order_by: Product.name asc
  limit: $limit
}
```

### `w17/domains/catalog/apis/public.proto` — facade (layer 3)

```protobuf
syntax = "proto3";
package w17.catalog.api.public;

import "w17/schema.proto";
import "w17/api.proto";

// --- Projections over the model (same as Experiment A) -----------------------
message Product {
  option (w17.schema.projection) = {
    from: "w17.catalog.products.Product"
  };
}

message CreateProductRequest {
  option (w17.schema.projection) = {
    from: "w17.catalog.products.Product"
  };
}

message UpdateProductRequest {
  option (w17.schema.projection) = {
    from:    "w17.catalog.products.Product"
    partial: true
  };
}

// --- The facade itself -------------------------------------------------------
// Mandatory layer. Typically a thin passthrough to the standard service.
service CatalogPublicAPI {
  option (w17.api.config) = {
    // gateway: REST + gRPC-web, transport: proto-over-HTTP
  };

  // Passthrough — one line; standard service uses the same request/response.
  rpc CreateProduct (CreateProductRequest) returns (Product) {
    option (w17.api.forward) = { to: "w17.catalog.service.CatalogService" };
  }

  rpc UpdateProduct (UpdateProductRequest) returns (Product) {
    option (w17.api.forward) = { to: "w17.catalog.service.CatalogService" };
  }

  // Reshape — different input shape; mapped on the wire.
  rpc GetProduct (GetProductRequest) returns (Product) {
    option (w17.api.forward) = {
      to: "w17.catalog.service.CatalogService.GetProductBySlug"
      map: "slug: $input.slug"
    };
  }
}

message GetProductRequest {
  string slug = 1;
}
```

### `w17/domains/catalog/events/product_events.proto`

```protobuf
syntax = "proto3";
package w17.catalog.events;

import "w17/event.proto";
import "google/protobuf/empty.proto";

// External async contract of the `catalog` domain.
// Reference into module `products` is OK — events live at the domain root
// intentionally.
service CatalogEvents {
  option (w17.event.config) = {
    transport: "nats"
    subject_prefix: "catalog."
  };

  rpc OnProductCreated (w17.catalog.products.Product) returns (google.protobuf.Empty) {
    option (w17.event.trigger) = {
      on:     [CREATE]
      source: "w17.catalog.products.Product"
    };
  }

  rpc OnProductStockChanged (w17.catalog.products.Product) returns (google.protobuf.Empty) {
    option (w17.event.trigger) = {
      on:     [UPDATE]
      source: "w17.catalog.products.Product"
      when:   "old.stock_quantity != new.stock_quantity"
    };
  }
}
```

### What Experiment D produces

A single developer writes ~120 lines of declarative code. Out of the compiler falls:

- **1 SQL migration** (tables `products` + `categories`, indexes, FK, soft-delete columns)
- **1 Storage service (Go binary)** — 5+ gRPC methods
  (Create/Get/Update/Delete/List/GetBySlug/ListActiveByCategory)
- **1 Standard service skeleton (Go)** — developer fills in handler bodies; DI and
  event emission are wired automatically
- **1 Facade service** — fully generated, including REST gateway
- **1 Event system** — auto-emitted events over NATS; no `emit()` calls in code
- **OpenAPI + AsyncAPI** — for frontend and subscribers
- **TS client** — proto-over-HTTP, with type safety and client-side validation
- **Django-style admin** (when UI metadata is added to the model)

Hand-written code: **only the handler bodies in `CatalogService`** — about 3 functions,
10-20 lines each.

```
================================================================================
 OPEN QUESTIONS  (to resolve before Phase 2)
================================================================================
```

1. **Annotation inheritance.** If `(w17.validate) = { min_len: 1 }` is on the model,
   should it propagate into a `CreateProductRequest` projection automatically?
   Tentative: **yes**, with the option to override at the projection level.

2. **Field numbers.** Does a projection reuse the model's field numbers (proto
   compatibility) or assign its own? Tentative: **same numbers as in the model**;
   added fields take 100+.

3. **Boundary vs. Mapping DSL (Phase 4).** Projection = reshape of **one** model;
   mapping = composition across **multiple** models/services. Is the line sharp
   enough? When does `schema.projection` end and `mapping` begin?

4. **Partial update semantics.** Does `partial: true` mean every field optional?
   FieldMask? The proto `optional` keyword? Must be settled before Phase 2.

5. **Cross-domain referencing.** Can `api/public.proto` in domain A reference a
   model in domain B? Proposal: **no** — it has to go through domain B's facade,
   and the compiler enforces it statically. Outstanding: what does the syntax
   look like for "call a foreign facade" from inside a service?

6. **`common/` discipline.** What belongs and does not belong in `common/`?
   Proposal: **value types (Money, Address, DateRange, enums)**, optionally also
   **template models** (each domain materializes its own table). Forbidden:
   storage logic, events, facades.

7. **Facade forwarding DSL.** `(w17.api.forward)` is a first stab. For non-trivial
   mappings (aggregation over several service calls) it overlaps with the Mapping
   DSL from Phase 4 — the two need to be reconciled.
